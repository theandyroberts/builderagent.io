If you've read more than two essays about AI in the last six months, you've probably noticed a problem: the word "agent" is doing a lot of work, and not all of it is the same work.

Some people use "agent" to mean a chatbot. Some use it to mean a tool-using LLM. Some use it to mean a fully autonomous system that runs forever. The disagreement isn't just academic — it shapes what you build, what you buy, and what you trust an AI to do unsupervised.

Here's the working definition we use at BuilderAgent, and the one we think is most useful for people actually shipping things.

## The four-part definition

**An AI agent is a system that:**

1. **Takes a goal** — usually in plain language, sometimes structured.
2. **Plans** — breaks the goal into steps, ordered, with dependencies.
3. **Acts** — executes the steps using tools (web, APIs, code, integrations).
4. **Loops** — observes the results, revises the plan, keeps going.

If a system has all four, it's an agent. Drop any one and you have something simpler.

A chatbot has step 3 (sort of) but not 4. A workflow tool has step 3 and 4 but not 2 (you wrote the plan). A "deep research" feature has steps 2 and 3 but stops after one pass — it's a one-shot agent, not a continuous one.

## The continuous loop is what changes the game

Most products marketed as "agents" today are actually one-shot. You give them a task, they run, they return, you call them again next time.

A continuous agent is different. It runs against a long-running goal — "monitor competitor pricing", "track legislation in this state", "watch this support queue" — and it keeps working. The loop is the product.

This is the part most builders underestimate. Building one-shot AI features is a few weeks of work. Building agents that run for months without falling over is a year of platform engineering: state management, model routing, error recovery, memory persistence, observability. That's why platforms exist.

## The OpenClaw model

OpenClaw — the open-source agent runtime BuilderAgent and InstaClaw.bot are both built on — encodes this four-part definition directly:

- A **goal** is the persistent objective the agent runs against.
- A **plan** is generated by the LLM, revised every loop.
- **Tools** are skills the agent can invoke (built-in + custom).
- The **loop** runs continuously, with checkpointing so the agent can resume across model upgrades and infrastructure events.

The reason we like this model is that it doesn't conflate "what the agent is" with "what model it runs". The same goal, plan, and loop should run on Claude 4, GPT-5, or whatever frontier model arrives next month. Models are interchangeable parts. Agents are durable.

## What's *not* an agent (even if marketing says so)

A few things to be allergic to:

- **A system prompt that says "you are an agent"** — that's roleplay, not architecture.
- **A wrapper that calls an API once and returns** — that's a function. Useful, but not an agent.
- **A multi-step workflow with no replanning** — that's a pipeline. Also useful. Also not an agent.

The difference shows up the moment something unexpected happens. A pipeline crashes; an agent reroutes. A function returns "I don't know"; an agent goes and finds out.

## Why this definition matters for builders

If you're choosing between building agent features and using an agent platform, the question to ask is: **how much of the four-part loop am I going to own?**

If you only need the "act" step — call this API, format the output — a thin LLM wrapper is fine. If you need plan + act + loop running for months without you, you're building infrastructure, not a feature. That's the gap [BuilderAgent](/) fills.

The point isn't to gatekeep the word "agent". The point is to know what you're committing to when you say it.
